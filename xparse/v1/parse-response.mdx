The Document Parsing API returns a unified JSON structure containing the document's Markdown representation, structured element list, page metadata, and more. This guide details all response fields.

## Response Overview

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "schema_version": "1.3.0",
    "file_id": "doc_7f3a2b",
    "job_id": "job_x9k2m",
    "success_count": 5,
    "metadata": { ... },
    "markdown": "# Title\n\nBody content...",
    "elements": [ ... ],
    "title_tree": [ ... ],
    "pages": [ ... ],
    "summary": { ... }
  }
}
```

### Top-Level Fields

| Field | Type | Description |
|------|------|------|
| `code` | int | Status code, 200 indicates success |
| `message` | string | Status message |
| `data` | object | Parsing result data |

### data Fields

| Field | Type | Description |
|------|------|------|
| `schema_version` | string | Data structure version, currently `"1.3.0"` |
| `file_id` | string | Unique file identifier |
| `job_id` | string | Unique job identifier |
| `success_count` | int | Number of successfully parsed pages (billing basis) |
| `metadata` | object | File metadata |
| `markdown` | string | Markdown representation of the document |
| `elements` | array | List of document elements |
| `title_tree` | array | Document table of contents (requires `title_tree` capability) |
| `pages` | array | Page metadata list (requires `pages` capability) |
| `summary` | object | Processing time statistics |

## metadata (File Metadata)

```json
{
  "filename": "document.pdf",
  "filetype": "application/pdf",
  "page_count": 10,
  "data_source": {
    "record_locator": {
      "protocol": "file",
      "remote_file_path": "/path/to/document.pdf"
    },
    "url": "file:///path/to/document.pdf"
  }
}
```

| Field | Type | Description |
|------|------|------|
| `filename` | string | File name |
| `filetype` | string | File MIME type |
| `page_count` | int | Total page count |
| `data_source` | object | Data source details including protocol and path |

## elements (Document Elements)

`elements` is the core of parsing results. Each element represents a structured unit in the document (heading, paragraph, table, image, etc.).

<Note>
  **GUI Engine Differences**: When using `force_engine: "textin_gui"`, the elements structure differs, including GUI-specific fields and element types. See [GUI Recognition Engine Notes](#gui-recognition-engine-notes).
</Note>

### Basic Structure

```json
{
  "element_id": "el_001",
  "type": "Title",
  "text": "Chapter 1 Overview",
  "page_number": 1,
  "coordinates": [0.100000, 0.120000, 0.320000, 0.120000, 0.320000, 0.160000, 0.100000, 0.160000],
  "metadata": {
    "category_depth": 0,
    "children_ids": ["el_002", "el_003"],
    "is_continuation": false,
    "data_source": {
      "record_locator": {
        "protocol": "file",
        "remote_file_path": "/path/to/document.pdf"
      },
      "url": "file:///path/to/document.pdf"
    }
  }
}
```

### Base Fields

| Field | Type | Description |
|------|------|------|
| `element_id` | string | Unique element identifier |
| `type` | string | Element type (see type table below) |
| `sub_type` | string | Element subtype (optional) for further classification. E.g., `Image` may have subtypes like `stamp`, `qrcode`, `barcode`, `chart`; `Table` may have `bordered`, `borderless` subtypes |
| `text` | string | Element text content |
| `page_number` | int | Page number (starting from 1) |
| `coordinates` | array | Four-point coordinates, normalized to \[0, 1\], order: top-left, top-right, bottom-right, bottom-left |
| `metadata` | object | Element metadata |
| `objects` | array | List of inline objects within the element (requires `include_inline_objects`, see below) |
| `table_structure` | object | Table structure details (`Table` elements only, requires `include_table_structure`, see below) |
| `char_details` | array | Character-level details (requires `include_char_details`, see below) |
| `image_data` | object | Image data (`Image` elements only, requires `include_image_data`, see below) |

<Tip>
  **GUI Engine Exclusive Fields**: When using the GUI engine, elements also include `interactivity` (whether interactive) and `description` (semantic description) fields. [See details](#gui-recognition-engine-notes)
</Tip>

### Element Types

| Type | Description |
|------|------|
| `Title` | Heading |
| `NarrativeText` | Body paragraph |
| `ListItem` | List item |
| `Table` | Table |
| `TableCaption` | Table caption |
| `Image` | Image |
| `FigureCaption` | Figure caption |
| `Formula` | Mathematical formula |
| `Chemical` | Chemical molecular formula |
| `Header` | Page header |
| `Footer` | Page footer |
| `PageNumber` | Page number |
| `PageBreak` | Page break |
| `CodeSnippet` | Code snippet |
| `UncategorizedText` | Uncategorized text |

<Note>
  The element types above apply to the document parsing engine. **The GUI engine uses different element types** (such as button, input, checkbox, etc.). See [UI Element Types](#ui-element-types-type).
</Note>

### metadata Fields

| Field | Type | Description |
|------|------|------|
| `parent_id` | string | Parent element ID (requires `include_hierarchy`) |
| `children_ids` | array | List of child element IDs (requires `include_hierarchy`) |
| `category_depth` | int | Nesting depth of same-type elements (e.g., 0 for Title indicates level-1 heading, 1 for level-2) |
| `ref_element_id` | string | Referenced element ID, e.g., association between image/table and its caption (requires `include_hierarchy`) |
| `is_continuation` | bool | Whether this element continues from a previous page |
| `continuation_of` | string | When `is_continuation=true`, points to the previous element ID being continued |
| `has_inline_objects` | bool | Whether contains inline objects (requires `include_inline_objects`) |
| `inline_object_types` | array | List of inline object types, e.g., `["formula", "handwriting"]` |
| `width` | int | Image width (`Image` elements only) |
| `height` | int | Image height (`Image` elements only) |
| `data_source` | object | Data source details |

## Coordinate System

Coordinates use normalized four-point representation, with each coordinate value in the `\[0, 1\]` range, representing the proportion relative to page width and height.

```
coordinates: [x1, y1, x2, y2, x3, y3, x4, y4]
```

The four points follow this order:

```
(x1,y1) -------- (x2,y2)
   |                |
   |  Element Area  |
   |                |
(x4,y4) -------- (x3,y3)
```

Coordinate values are rounded to six decimal places in the \[0, 1\] range, representing proportions relative to page dimensions. To convert normalized coordinates to pixel coordinates, multiply by the actual page dimensions:

```python
# Assuming page dimensions: 595px width, 842px height (A4)
page_width, page_height = 595, 842
pixel_x1 = coordinates[0] * page_width
pixel_y1 = coordinates[1] * page_height
```

## Table Structure (table_structure)

When the `include_table_structure` capability is enabled, elements with type `Table` include a `table_structure` field.

```json
{
  "element_id": "el_010",
  "type": "Table",
  "text": "| Name | Age |\n|---|---|\n| John | 28 |",
  "table_structure": {
    "rows": 2,
    "cols": 2,
    "cells": [
      {
        "cell_id": "tbl_001_r1_c1",
        "row": 1,
        "col": 1,
        "row_span": 1,
        "col_span": 1,
        "content_type": "text",
        "text": "Name",
        "coordinates": [0.100000, 0.300000, 0.300000, 0.300000, 0.300000, 0.340000, 0.100000, 0.340000]
      },
      {
        "cell_id": "tbl_001_r1_c2",
        "row": 1,
        "col": 2,
        "row_span": 1,
        "col_span": 1,
        "content_type": "text",
        "text": "Age",
        "coordinates": [0.300000, 0.300000, 0.500000, 0.300000, 0.500000, 0.340000, 0.300000, 0.340000]
      },
      {
        "cell_id": "tbl_001_r2_c1",
        "row": 2,
        "col": 1,
        "row_span": 1,
        "col_span": 1,
        "content_type": "text",
        "text": "John",
        "coordinates": [0.100000, 0.340000, 0.300000, 0.340000, 0.300000, 0.380000, 0.100000, 0.380000]
      },
      {
        "cell_id": "tbl_001_r2_c2",
        "row": 2,
        "col": 2,
        "row_span": 1,
        "col_span": 1,
        "content_type": "text",
        "text": "28",
        "coordinates": [0.300000, 0.340000, 0.500000, 0.340000, 0.500000, 0.380000, 0.300000, 0.380000]
      }
    ]
  }
}
```

### Cell Fields

| Field | Type | Description |
|------|------|------|
| `cell_id` | string | Unique cell identifier |
| `row` | int | Row index (starting from 1) |
| `col` | int | Column index (starting from 1) |
| `row_span` | int | Number of rows spanned |
| `col_span` | int | Number of columns spanned |
| `content_type` | string | Content type: `text`, `formula`, `image`, `mixed` |
| `text` | string | Cell text content |
| `coordinates` | array | Cell four-point coordinates |
| `image_datas` | array | Array of image data (when `content_type` is `image`, requires `include_image_data`) |
| `objects` | array | List of embedded objects in cell (requires `include_inline_objects`) |
| `char_details` | array | Character-level details (requires `include_char_details`) |

## Image Data (image_data)

When the `include_image_data` capability is enabled, elements with type `Image` include an `image_data` field.

```json
{
  "element_id": "el_020",
  "type": "Image",
  "text": "",
  "image_data": {
    "image_url": "https://example.com/images/xxx.png",
    "mime_type": "image/png",
    "base64": "iVBORw0KGgoAAAANSUhEUg..."
  }
}
```

| Field | Type | Description |
|------|------|------|
| `image_url` | string | Image access URL |
| `mime_type` | string | Image MIME type |
| `base64` | string | Image Base64 encoding (optional) |

## Character Details (char_details)

When the `include_char_details` capability is enabled, elements include a `char_details` field providing character-level coordinates and recognition confidence.

```json
{
  "element_id": "el_001",
  "type": "Title",
  "text": "Overview",
  "char_details": [
    {
      "index": 0,
      "text": "O",
      "coordinates": [0.100000, 0.120000, 0.140000, 0.120000, 0.140000, 0.160000, 0.100000, 0.160000],
      "recognition": {
        "confidence": 0.999,
        "candidates": [
          {"text": "O", "confidence": 0.999},
          {"text": "0", "confidence": 0.001}
        ]
      }
    },
    {
      "index": 1,
      "text": "v",
      "coordinates": [0.140000, 0.120000, 0.180000, 0.120000, 0.180000, 0.160000, 0.140000, 0.160000],
      "recognition": {
        "confidence": 0.998,
        "candidates": [
          {"text": "v", "confidence": 0.998},
          {"text": "u", "confidence": 0.002}
        ]
      }
    }
  ]
}
```

| Field | Type | Description |
|------|------|------|
| `index` | int | Character position index in text |
| `text` | string | Character text |
| `coordinates` | array | Character four-point coordinates |
| `recognition.confidence` | float | Recognition confidence (0-1) |
| `recognition.candidates` | array | Candidate recognition results |

## Inline Objects (objects)

When the `include_inline_objects` capability is enabled, elements containing inline objects return an `objects` field, identifying formulas, handwriting, checkboxes, and other inline elements within text.

```json
{
  "element_id": "el_004",
  "type": "NarrativeText",
  "text": "Given the penalty function $f(x)=x^2+1$.",
  "objects": [
    {
      "object_id": "obj_001",
      "type": "formula",
      "text": "$f(x)=x^2+1$",
      "text_range": [28, 41],
      "coordinates": [0.220000, 0.480000, 0.300000, 0.480000, 0.300000, 0.540000, 0.220000, 0.540000],
      "metadata": {
        "display_mode": "inline"
      }
    }
  ]
}
```

| Field | Type | Description |
|------|------|------|
| `object_id` | string | Unique object identifier |
| `type` | string | Object type: `formula`, `handwriting`, `checkbox`, `image` |
| `sub_type` | string | Object subtype (optional) |
| `text` | string | Object text content |
| `text_range` | array | Object position in parent element text `[start, end)`, 0-based half-open interval |
| `coordinates` | array | Object four-point coordinates |
| `image_data` | object | Image data (when `type` is `image`) |
| `metadata` | object | Object metadata, e.g., formula `display_mode` (`inline` or `display`) |

## Title Tree (title_tree)

When the `title_tree` capability is enabled, returns the document's hierarchical outline structure:

```json
{
  "title_tree": [
    {
      "element_id": "el_001",
      "title": "Chapter 1 Overview",
      "level": 1,
      "page_number": 1,
      "children": [
        {
          "element_id": "el_005",
          "title": "1.1 Background",
          "level": 2,
          "page_number": 1,
          "children": []
        },
        {
          "element_id": "el_010",
          "title": "1.2 Objectives",
          "level": 2,
          "page_number": 2,
          "children": []
        }
      ]
    }
  ]
}
```

| Field | Type | Description |
|------|------|------|
| `element_id` | string | ID of the corresponding Title element |
| `title` | string | Title text |
| `level` | int | Title level, 1 is highest (level-1 heading) |
| `page_number` | int | Page number |
| `children` | array | List of nested child title nodes |

## Page Information (pages)

When the `pages` capability is enabled, returns metadata for each page:

```json
{
  "pages": [
    {
      "page_number": 1,
      "page_width": 1576,
      "page_height": 1683,
      "page_image_url": "https://example.com/page-1.jpg",
      "element_ids": ["el_001", "el_002", "el_003"],
      "dpi": 144,
      "angle": 0,
      "status": "Success"
    }
  ]
}
```

| Field | Type | Description |
|------|------|------|
| `page_number` | int | Page number (starting from 1) |
| `page_width` | number | Page width (pixels) |
| `page_height` | number | Page height (pixels) |
| `page_image_url` | string | Rendered page image URL |
| `element_ids` | array | List of element IDs on this page, ordered by default reading order |
| `dpi` | int | DPI used for converting page to image |
| `angle` | number | Page rotation angle (0 degrees for normal reading orientation, clockwise rotation) |
| `status` | string | Page processing status |

| Field | Type | Description |

| `duration_ms` | number | Total processing time (milliseconds) |

## Error Responses

When `code` is not 200, the request failed. Error responses may include a `location` field to pinpoint where the error occurred:

```json
{
  "code": 40004,
  "message": "Parameter error, please check technical documentation and verify parameters",
  "location": {
    "stage": "parse",
    "page_number": 3,
    "element_id": "el_045"
  }
}
```

| Field | Type | Description |
|------|------|------|
| `stage` | string | Error stage |
| `page_number` | int | Error page number |
| `element_id` | string | Error element ID |

### Common Error Codes

| Error Code | HTTP Status | Description |
|--------|-------------|------|
| 40101 | 200 | x-ti-app-id or x-ti-secret-code is empty |
| 40102 | 200 | x-ti-app-id or x-ti-secret-code is invalid, authentication failed |
| 40103 | 200 | Client IP not in whitelist |
| 40003 | 200 | Insufficient balance, please recharge |
| 40004 | 200 | Parameter error, please check technical documentation |
| 40007 | 200 | Robot does not exist or is not published |
| 40008 | 200 | Robot not activated, please activate from marketplace |
| 40301 | 200 | Image type not supported |
| 40302 | 200 | File size exceeds limit, maximum 500MB |
| 40303 | 200 | File type not supported, API returns detected file type |
| 40304 | 200 | Image dimensions out of range, width and height must be 20-20000 pixels (aspect ratio &lt;2) or 20-10000 pixels (other ratios) |
| 40305 | 200 | Recognition file not uploaded |
| 40306 | 200 | QPS limit exceeded |
| 40307 | 200 | Daily free quota exhausted |
| 40422 | 200 | File corrupted |
| 40423 | 200 | Password required or incorrect password |
| 40424 | 200 | Page number out of range |
| 40425 | 200 | Input file format not supported |
| 40427 | 200 | Input DPI not in allowed list (72, 144, 216) |
| 40428 | 200 | Office file conversion failed or timed out |
| 40429 | 200 | Unsupported engine |
| 50207 | 200 | Partial pages failed to parse |
| 40400 | 200 | Invalid request URL, please verify |
| 30203 | 200 | Base service failure, please retry later |
| 500 | 200 | Internal server error |

<Tip>
  When encountering errors, use the `x-request-id` from response headers to contact technical support for troubleshooting.
</Tip>

## GUI Recognition Engine Notes

When using `force_engine: "textin_gui"`, the response structure is similar to normal document parsing, but the `type` field values and `metadata` field content in `elements` differ.

<Note>
  The GUI engine only supports image format input (JPEG, PNG, GIF, WebP), file size up to 10MB, and does not support document formats like PDF. See [Parse Configuration - force_engine](/xparse/v1/parse-config#force_engine) for configuration details.
</Note>

### Basic Structure

```json
{
  "element_id": "el_001",
  "type": "button",
  "text": "Login",
  "page_number": 1,
  "coordinates": [0.450000, 0.600000, 0.550000, 0.600000, 0.550000, 0.640000, 0.450000, 0.640000],
  "metadata": {
    "parent_id": "el_000",
    "children_ids": [],
    "is_continuation": false,
    "interactivity": true,
    "description": "Login button",
    "data_source": {
      "record_locator": {
        "protocol": "file",
        "remote_file_path": "/path/to/screenshot.png"
      },
      "url": "file:///path/to/screenshot.png"
    }
  }
}
```

### Base Fields

Same as the document parsing engine, includes the following fields:

| Field | Type | Description |
|------|------|------|
| `element_id` | string | Unique element identifier, same as other engines |
| `type` | string | **UI element type** (see type table below, different from document parsing engine) |
| `text` | string | Visible text/label content of the element, same as other engines |
| `page_number` | int | Page number (starting from 1), same as other engines |
| `coordinates` | array | Coordinate information, same as other engines |
| `metadata` | object | Element metadata (**includes GUI-specific fields**, see below) |

### UI Element Types (type)

Unlike the document parsing engine, the GUI engine returns `type` as UI component types:

| Element Type | Description | Examples |
|---------|------|------|
| `button` | Button | Submit button, close button |
| `input` | Input field | Text box, password field, search box |
| `checkbox` | Checkbox | Multi-select box |
| `dropdown` | Dropdown menu | Selector, dropdown list |
| `link` | Link | Hyperlink, navigation link |
| `tab` | Tab | Tab control |
| `text` | Static text | Label text, description text |
| `label` | Label | Form label, category tag |
| `title` | Title | Window title, section title |
| `icon` | Icon | Function icon, status icon |
| `image` | Image | Thumbnail, background image |
| `menu` | Menu | Navigation menu, context menu |
| `navigation` | Navigation bar | Top navigation, sidebar navigation |
| `toolbar` | Toolbar | Action toolbar |
| `statusbar` | Status bar | Bottom status bar |

### metadata Fields

The GUI engine's `metadata` includes the same common fields as the document parsing engine (parent_id, children_ids, is_continuation, data_source, etc.), plus these exclusive fields:

| Field | Type | Description |
|------|------|------|
| `interactivity` | bool | Whether interactive |
| `description` | string | Semantic description of the element |
