The Document Parsing API supports custom parsing behavior through the `config` parameter. This guide details all available configuration options.

## Configuration Structure Overview

```json
{
  "document": {
    "password": "example-pdf-password"
  },
  "capabilities": {
    "include_hierarchy": true,
    "include_inline_objects": false,
    "include_char_details": false,
    "include_image_data": false,
    "include_table_structure": false,
    "pages": false,
    "title_tree": false,
    "table_view": "html",
    "remove_watermark": false,
    "crop_dewarp": false
  },
  "scope": {
    "page_range": "1-5"
  },
  "config": {
    "force_engine": "textin",
    "engine_params": {
      "parse_mode": "vlm",
      "formula_level": 0,
      "image_output_type": "url",
      "recognize_chemical": true
    }
  }
}
```

## document (Document Settings)

Configure document-specific processing parameters.

```json
{
  "document": {
    "password": "example-pdf-password"
  }
}
```

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `password` | string | No | Password for encrypted documents (e.g., password-protected PDFs) |

**Use cases**:

- Process password-protected PDF documents
- Ensure encrypted documents parse correctly

## capabilities (Parsing Strategy & Output Format)

Control the level of detail and format of returned data. Enabling more capabilities increases processing time and response size.

### include\_hierarchy

Whether to return hierarchical relationships and associations between elements.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `include_hierarchy` | boolean | `true` | When enabled, returns parent-child relationships and reference relationships between elements |

**Fields returned when enabled**:

- `parent_id`: Parent element ID
- `children_ids`: List of child element IDs
- `ref_element_id`: Referenced element ID (e.g., association between images/tables and their captions)

**Use cases**:

- Build structured relationship graphs of document content
- Understand hierarchical relationships between elements
- Track relationships between headings and content

```json
{
  "capabilities": {
    "include_hierarchy": true
  }
}
```

### include\_inline\_objects

Whether to return fine-grained inline objects.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `include_inline_objects` | boolean | `false` | When enabled, text elements return contained fine-grained objects |

**Supported inline object types**:

- `formula`: Mathematical formulas (LaTeX format)
- `handwriting`: Handwritten content
- `checkbox`: Checkboxes
- `image`: Embedded images

**Use cases**:

- Precisely locate and extract formulas
- Identify handwritten signatures or annotations
- Process checkboxes in forms
- Extract embedded images from text paragraphs

```json
{
  "capabilities": {
    "include_inline_objects": true
  }
}
```

**Response example**:

```json
{
  "element_id": "el_004",
  "type": "NarrativeText",
  "text": "Given the penalty function $f(x)=x^2+1$.",
  "objects": [
    {
      "object_id": "obj_001",
      "type": "formula",
      "text": "$f(x)=x^2+1$",
      "text_range": [7, 20],
      "coordinates": [0.220000, 0.480000, 0.300000, 0.480000, 0.300000, 0.540000, 0.220000, 0.540000],
      "metadata": {
        "display_mode": "inline"
      }
    }
  ]
}
```

### include\_char\_details

Whether to return character-level detailed information.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `include_char_details` | boolean | `false` | When enabled, returns detailed information for each character in the text |

**Returned character information includes**:

- Character coordinates
- Recognition confidence
- Candidate character list

**Use cases**:

- Precise character-level positioning
- Assess recognition quality
- Handle low-confidence characters
- Implement character-level error correction

```json
{
  "capabilities": {
    "include_char_details": true
  }
}
```

**Response example**:

```json
{
  "char_details": [
    {
      "index": 0,
      "text": "O",
      "coordinates": [0.100000, 0.120000, 0.140000, 0.120000, 0.140000, 0.160000, 0.100000, 0.160000],
      "recognition": {
        "confidence": 0.999,
        "candidates": [
          {"text": "O", "confidence": 0.999},
          {"text": "0", "confidence": 0.001}
        ]
      }
    }
  ]
}
```

### include\_image\_data

Whether to return image data.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `include_image_data` | boolean | `false` | When enabled, image elements return complete image data |

**Returned image information includes**:

- Image URL
- MIME type
- Base64 encoding (optional)

**Use cases**:

- Download or display images from documents
- Perform secondary processing on images
- Obtain Base64 encoding for embedding

```json
{
  "capabilities": {
    "include_image_data": true
  }
}
```

**Response example**:

```json
{
  "element_id": "el_020",
  "type": "Image",
  "image_data": {
    "image_url": "https://example.com/images/xxx.png",
    "mime_type": "image/png",
    "base64": "iVBORw0KGgoAAAANSUhEUg..."
  }
}
```

### include\_table\_structure

Whether to return detailed structured information for tables.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `include_table_structure` | boolean | `false` | When enabled, returns table structure in JSON format with rows, columns, and detailed cell information |

**Returned table structure includes**:

- Row and column count
- Cell position (row, column)
- Cell span information (rowspan, colspan)
- Cell content type (text, formula, image, mixed)
- Cell coordinates

**Use cases**:

- Programmatically process table data
- Extract precise cell positions
- Handle complex tables (merged cells)
- Identify formulas or images in table cells

```json
{
  "capabilities": {
    "include_table_structure": true
  }
}
```

**Response example**:

```json
{
  "element_id": "el_010",
  "type": "Table",
  "table_structure": {
    "rows": 2,
    "cols": 2,
    "cells": [
      {
        "cell_id": "tbl_001_r1_c1",
        "row": 1,
        "col": 1,
        "row_span": 1,
        "col_span": 1,
        "content_type": "text",
        "text": "Name",
        "coordinates": [0.100000, 0.300000, 0.300000, 0.300000, 0.300000, 0.340000, 0.100000, 0.340000]
      }
    ]
  }
}
```

### pages

Whether to return page metadata list.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `pages` | boolean | `false` | When enabled, returns detailed information for each page |

**Returned page information includes**:

- Page number
- Page width and height
- Rotation angle
- Rendered image URL (`page_image_url`)
- List of contained elements (`element_ids`)
- DPI
- Processing status

**Use cases**:

- Organize document content by page
- Obtain page preview images
- Understand physical page properties (dimensions, DPI)
- Locate which page an element appears on

```json
{
  "capabilities": {
    "pages": true
  }
}
```

**Response example**:

```json
{
  "pages": [
    {
      "page_number": 1,
      "page_width": 1576,
      "page_height": 1683,
      "page_image_url": "https://example.com/page-1.jpg",
      "element_ids": ["el_001", "el_002", "el_003"],
      "dpi": 144,
      "angle": 0,
      "status": "Success"
    }
  ]
}
```

### title\_tree

Whether to return a hierarchical title tree (table of contents).

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `title_tree` | boolean | `false` | When enabled, returns the document's hierarchical outline structure |

**Returned outline information includes**:

- Title text
- Title level (1 is the highest level)
- Page number
- Nested child titles

**Use cases**:

- Generate document navigation
- Organize content by section
- Understand document outline structure

```json
{
  "capabilities": {
    "title_tree": true
  }
}
```

**Response example**:

```json
{
  "title_tree": [
    {
      "element_id": "el_001",
      "title": "Chapter 1 Overview",
      "level": 1,
      "page_number": 1,
      "children": [
        {
          "element_id": "el_005",
          "title": "1.1 Background",
          "level": 2,
          "page_number": 1,
          "children": []
        }
      ]
    }
  ]
}
```

### table\_view

Table output format in Markdown.

| Field | Type | Default | Options | Description |
| --- | --- | --- | --- | --- |
| `table_view` | string | `"html"` | `markdown`, `html` | Controls table rendering format in the Markdown field |

**Format comparison**:

**Markdown format** (`table_view: "markdown"`):

```markdown
| Name | Age |
|---|---|
| John | 28 |
```

**HTML format** (`table_view: "html"`):

```html
<table>
  <thead>
    <tr><th>Name</th><th>Age</th></tr>
  </thead>
  <tbody>
    <tr><td>John</td><td>28</td></tr>
  </tbody>
</table>
```

**Use cases**:

- Need clean Markdown table format
- Use HTML for complex table structures (merged cells)

Whether to apply watermark removal preprocessing.

| Field | Type | Default | Description |
| :-- | :-- | :-- | :-- |
| `remove_watermark` | boolean | `false` | When enabled, automatically detects and removes watermarks from documents |

**Use cases**:

- Remove watermark interference to improve recognition accuracy
- Obtain clean parsing results without watermarks

```json
{
  "capabilities": {
    "remove_watermark": true
  }
}
```

### crop\_dewarp

Whether to apply crop and dewarp preprocessing.

| Field | Type | Default | Description |
| :-- | :-- | :-- | :-- |
| `crop_dewarp` | boolean | `false` | When enabled, automatically detects and applies cropping and dewarping |

**Use cases**:

- Scanned documents with extra borders or page tilt

- Need straight, compact page images to improve layout analysis quality


## scope (Processing Range Control)

Control the page range for parsing to reduce unnecessary processing.

```json
{
  "scope": {
    "page_range": "1-5,10-15"
  }
}
```

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `page_range` | string | No | Page range, starting from 1, supports multiple closed intervals |

**Format specifications**:

- Single page: `"1"`
- Continuous pages: `"1-5"`
- Multiple ranges: `"1-2,3-4,5-10"`

<Tip>
  **Single File Page Limits**

  When file page count exceeds the request limit, the API returns an error. See limits and solutions below:

  | User Type | Sync API Limit | Async API Limit | Available Solutions |
  | --- | --- | --- | --- |
  | Free Users | ≤100 pages | ≤100 pages | ① Use `page_range` parameter for batch processing<br />② Upgrade to paid plan<br />③ Switch to [Async API](/api-reference/endpoint/xparse/v1/parse-async) |
  | Paid Users | ≤500 pages | ≤2000 pages | ① Use `page_range` parameter for batch processing<br />② For files over 500 pages, use [Async API](/api-reference/endpoint/xparse/v1/parse-async) |
  | Enterprise | ≤500 pages | ≤5000 pages | ① Use `page_range` parameter for batch processing<br />② For files over 500 pages, use [Async API](/api-reference/endpoint/xparse/v1/parse-async) |
</Tip>

**Use cases**:

- Process only specific pages of a document
- Reduce processing time and cost
- Batch process large documents

## config (Advanced Configuration)

Expert mode configuration for forcing specific parsing engines and engine parameters.

<Warning>
  **Use with caution**: These settings are for advanced users only. Improper configuration may degrade parsing quality or cause failures.
</Warning>

### force\_engine

Force a specific internal parsing engine.

| Field | Type | Options | Description |
| --- | --- | --- | --- |
| `force_engine` | string | `textin`, `textin_gui` | Force use of a specific parsing engine |

**Engine descriptions**:

- `textin`: TextIn proprietary engine, default option, best overall performance
- `textin_gui`: GUI recognition engine for desktop/mobile/web application interface screenshots

<Note>
  `textin_gui` only supports image formats (JPEG, PNG, GIF, WebP), file size up to 10MB, and has a different response structure. See [GUI Recognition Engine Notes](/xparse/v1/parse-response#gui-recognition-engine-notes)
</Note>

**Use cases**:

- Compare results across different engines
- Specific scenarios requiring specific engines
- Debugging and testing

```json
{
  "config": {
    "force_engine": "textin"
  }
}
```

### engine\_params

Engine-level custom parameters; different engines support different parameters.

| Field | Type | Description |
| --- | --- | --- |
| `engine_params` | object | Custom parameters passed to the parsing engine |

**Common parameter examples**:

```json
{
  "config": {
    "force_engine": "textin",
    "engine_params": {
      "parse_mode": "parse",
      "formula_level": 0,
      "image_output_type": "url",
      "recognize_chemical": true
    }
  }
}
```

| Parameter | Type | Options | Description |
| --- | --- | --- | --- |
| `parse_mode` | string | `auto`, `scan`, `parse`, `lite`, `vlm` | PDF parsing mode.<br />• `auto`: Engine auto-selects, widest applicability<br />• `scan`: Parse as images<br />• `parse`: Digital text only, fastest<br />• `lite`: Lightweight version, outputs tables and text only<br />• `vlm`: Vision-language model parsing mode |
| `formula_level` | int | `0`, `1` | Formula recognition level, 0 for standard, 1 for enhanced |
| `image_output_type` | string | `url`, `base64` | Image output type, `url` or `base64` |
| `recognize_chemical` | boolean | `true`, `false` | Extract chemical molecular structure formulas, only effective in `vlm` mode, results appear in element text field |

<Note>
  Different engines support different parameters. Contact technical support for specific parameter details.
</Note>

## Complete Configuration Examples

### Basic Configuration (Recommended)

Default configuration suitable for most scenarios:

```json
{
  "capabilities": {
    "include_hierarchy": true,
    "title_tree": true,
```

Return all available detailed information (increases processing time and data size):

```json
{
  "capabilities": {
    "include_hierarchy": true,
    "include_inline_objects": true,
```


## Performance Optimization Tips

<AccordionGroup>
  <Accordion title="Enable capabilities as needed" icon="toggle-on">
    Only enable necessary capability switches to avoid returning unnecessary data. For example, if you don't need character-level details, don't enable `include_char_details`.
  </Accordion>

  <Accordion title="Use page_range wisely" icon="file">
    For large documents, process a subset of pages first for testing, then process all pages once results are confirmed.
  </Accordion>

  <Accordion title="Use async for large files" icon="clock">
    For documents over 50 pages, use the async API to avoid HTTP timeouts.
  </Accordion>

  <Accordion title="Cache processing results" icon="database">
    Identical documents produce the same `file_id`, which can be used to implement result caching.
  </Accordion>
</AccordionGroup>
