## What is Extract?

Extract pulls specific data from documents using your JSON Schema definition. Get only the fields you need—invoice totals, form data, table rows—without processing the entire document.

```
Document + Schema → Extract → JSON
```

---

## Parse vs Extract

| Feature | Parse | Extract |
|---------|-------|---------|
| **Goal** | Full document content | Specific fields only |
| **Input** | File only | File + Schema |
| **Output** | Markdown + Elements | JSON matching schema |
| **Speed** | Slower | Faster |
| **Use case** | RAG, search, full text | Forms, invoices, structured data |

**Use Extract when:**
- You only need specific fields (not full content)
- Working with forms, invoices, receipts, contracts
- Data needs to match your database schema
- Speed matters (no need to parse everything)

**Use Parse when:**
- You need complete document content
- Building RAG or search systems
- Structure matters (headings, reading order)
- Want coordinates for all elements

See [Parse Overview](/xparse/parse/overview) for parsing details.

## Example: Invoice extraction

**Define schema:**
```json
{
  "type": "object",
  "properties": {
    "invoice_number": {
      "type": "string",
      "description": "Invoice number"
    },
    "total_amount": {
      "type": "number",
      "description": "Total amount including tax"
    },
    "items": {
      "type": "array",
      "description": "Invoice line items",
      "items": {
        "type": "object",
        "properties": {
          "description": {"type": "string"},
          "quantity": {"type": "number"},
          "price": {"type": "number"}
        }
      }
    }
  }
}
```

**Get structured output:**
```json
{
  "invoice_number": "INV-2024-001",
  "total_amount": 1250.00,
  "items": [
    {
      "description": "Product A",
      "quantity": 5,
      "price": 250.00
    }
  ]
}
```

## Use cases

### Invoice processing

Extract invoice number, date, vendor, line items, total → Import to accounting system

**Typical fields:**
- `invoice_number`, `invoice_date`
- `vendor_name`, `vendor_address`
- `items[]` (description, quantity, price)
- `subtotal`, `tax`, `total`

---

### Receipt automation

Extract merchant, date, items, total → Expense tracking

**Typical fields:**
- `merchant_name`, `date`, `time`
- `items[]` (name, price)
- `subtotal`, `tax`, `total`
- `payment_method`

---

### Form processing

Extract form field values → Populate database

**Typical fields:**
- Personal info (name, email, phone)
- Address (street, city, zip)
- Checkboxes, signatures
- Dates, amounts

---

### Contract metadata

Extract key terms without reading entire document → Contract management system

**Typical fields:**
- `parties[]` (names, roles)
- `effective_date`, `expiration_date`
- `contract_value`
- `termination_clause`

## Schema definition

Extraction uses [JSON Schema](https://json-schema.org/) to define data structure.

### Supported field types

| Type | Description | Example |
|------|-------------|---------|
| `string` | Text | `"John Doe"` |
| `number` | Numeric (int or float) | `123.45` |
| `integer` | Integer only | `42` |
| `enum` | Predefined choices | `["Paid", "Pending", "Overdue"]` |
| `object` | Nested fields | `{"name": "...", "address": "..."}` |
| `array` | List of items | `[{...}, {...}]` |

### Nesting limits

- Maximum 3 levels of nesting
- Maximum 100 leaf fields (lowest-level fields)

### Manual JSON

Write JSON Schema directly:

```json
{
  "type": "object",
  "properties": {
    "field_name": {
      "type": "string",
      "description": "What this field contains"
    }
  },
  "required": ["field_name"]
}
```

## Output format

Extraction returns:

### Extracted data

JSON matching your schema structure:
```json
{
  "field_name": "value",
  "nested_object": {
    "sub_field": "value"
  }
}
```

---

### Citations

Source locations for each extracted field:
- Page number
- Bounding box coordinates
- Original text

**Use for:**
- Verifying extraction accuracy
- Visual highlighting in UI
- Audit trails

---

### Confidence scores

Confidence level for each field (if available):
- High confidence: Model is certain
- Low confidence: May need manual review
